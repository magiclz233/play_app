# 同城伴玩 MVP 需求差距与开发计划

## 1. 当前需求理解

README 定义的核心业务闭环是：

1. 用户在小程序大厅浏览助教资料、相册、语音与价格。
2. 用户选择助教并在线支付全款，资金由平台托管。
3. 支付后平台/客服通过企业微信拉群，促成线下履约。
4. 服务完成后平台核销订单，并向助教钱包结算。
5. 用户可查看订单进度、申请退款、完成后评价。

参考截图补充的关键交互包括：

1. 助教列表展示可预约状态、距离、评价入口，并支持从列表或详情进入下单。
2. 下单页支持编辑地址/联系方式、选择服务日期与时间段。
3. 订单页展示订单进度、助教状态，并提供退款申请入口。
4. 退款页填写退款原因并提交，退款原路返回。
5. 订单完成后用户可评价，评价会影响助教公开评分。
6. 加钟/项目升级可作为后续增强，但需要新的补差价订单或订单变更模型。

## 2. 当前系统不足

### 2.1 用户下单流程不足

- 下单页目前默认使用“1 小时后”的时间，缺少可视化日期/时间段选择。
- 下单页没有填写服务地址、联系人电话/微信号，无法支撑客服线下对接。
- `OrderCreateDTO` 只接收预约时间、时长、备注，后端没有保存地址和联系方式。
- 订单详情页没有完整展示地址、联系方式、退款状态等关键履约信息。

### 2.2 订单履约闭环不足

- README 要求“客服拉群、平台核销、手动结算”，但后台原来只能查看订单列表。
- 用户确认完工原实现会直接给助教钱包入账，不符合“平台手动核销并结算”的担保逻辑。
- 订单状态日志表存在，但核心状态变更未统一写日志，后续追责和运营复盘困难。
- 服务过程记录表存在，但没有在拉群/开始/核销时生成或更新记录。

### 2.3 退款流程不足

- 用户端只有“取消订单”，没有独立的“申请退款”页面和原因输入。
- 已付款订单进入取消状态后，后台没有退款审核/处理入口。
- 退款当前只能作为状态变更占位，尚未对接微信退款接口。

### 2.4 评价流程不足

- 用户完成后可以评价，但评价服务会把订单直接置为已完成，与“平台结算后完成”逻辑存在冲突。
- 助教详情页暂未展示评价列表，只展示汇总评分。

### 2.5 助教列表体验不足

- 列表已有状态和价格，但距离字段为空，未展示“最近可约时间”。
- 搜索、排序、在线筛选目前主要是静态 UI，未真正传参到后端。
- 评价入口和评论数量缺失。

### 2.6 暂缓项

- 加钟、项目升级、补差价下单属于订单变更/二次支付模型，需要新增业务规则和支付链路；当前不纳入本轮 MVP。
- 真实 LBS 距离计算需要用户定位、助教服务坐标和隐私策略；当前先保留展示占位或固定文案。
- 微信企业客服名片/客服链接可配置化需要运营配置项；当前先以订单状态推进接口支撑人工客服流程。

## 3. 本轮开发范围

本轮只补齐 MVP 闭环，不扩展复杂交易模型。

### 3.1 后端开发需求

1. 扩展下单 DTO 与订单创建逻辑：
   - 接收并保存 `address`、`addressDetail`、`customerWechat`。
   - 根据开始时间和服务时长计算 `reserveTimeEnd`。

2. 补齐后台订单推进接口：
   - 已付款 -> 已拉群。
   - 已拉群 -> 服务中。
   - 服务中/已拉群 -> 平台核销完工。
   - 待结算 -> 平台结算放款。

3. 统一订单状态日志：
   - 创建、支付成功、拉群、服务开始、核销、结算、取消/退款申请写入 `order_status_logs`。

4. 补服务记录写入：
   - 拉群后创建 `service_records`。
   - 服务开始写入实际开始时间和地点。
   - 核销完工写入结束时间、实际时长、完工备注。

5. 调整评价与结算关系：
   - 用户评价只写评价与更新评分，不再替代平台结算。
   - 平台结算才将订单推进到 `80-已完成` 并入账钱包。

6. 退款申请：
   - 用户对已付款/已拉群/服务中/待确认订单可提交退款原因。
   - 后端先将订单置为 `100-申请取消` 并记录退款金额；真实微信退款后续对接。

### 3.2 前端开发需求

1. 下单页：
   - 新增地址、微信/手机号输入。
   - 新增日期和时间段选择，参考截图采用底部时间段面板。
   - 下单时提交地址、联系方式、开始时间、时长。

2. 订单列表页：
   - Tab 调整为全部、待支付、待服务、服务中、已完成。
   - 根据状态展示“去支付”“申请退款”“去评价”等按钮。
   - 订单卡片显示服务时间、金额、当前进度。

3. 订单详情页：
   - 展示地址、联系方式、服务时间、状态说明。
   - 增加申请退款入口。
   - 状态说明与后台推进后的 20/30/50/70/80 保持一致。

4. 新增退款页：
   - 展示订单金额与状态。
   - 输入退款原因并提交。
   - 提交后回到订单列表或详情刷新状态。

5. 管理后台订单页：
   - 对不同状态展示对应操作按钮：标记拉群、服务开始、核销完工、结算放款。
   - 操作成功后刷新订单列表。

## 4. 验收标准

1. 用户可以从助教详情进入下单页，选择时间段，填写地址和联系方式并创建订单。
2. Mock 支付后订单进入 `20-已付款/待拉群`。
3. 管理后台可以将订单依次推进到 `30-已拉群`、`50-服务中`、`70-待结算/待评价`、`80-已完成`。
4. 平台结算后助教钱包余额增加，钱包流水可查询。
5. 用户可以对可退款状态订单提交退款申请，订单进入 `100-申请取消`。
6. 用户可以在完成节点提交评价，助教评分更新。
7. 后端编译通过，前端构建通过。

## 5. 实施顺序

1. 后端补下单字段、状态推进、日志、服务记录与结算逻辑。
2. 后端补退款申请接口并修正评价状态逻辑。
3. 前端改下单页和新增退款页。
4. 前端改订单列表、订单详情、管理后台。
5. 运行后端编译与前端构建，修复发现的问题。

## 6. 设计原则约束

- KISS：复用现有订单状态、服务记录、钱包和日志表，不新增复杂表结构。
- YAGNI：暂缓加钟、升级、真实 LBS、微信真实退款等非当前闭环必需能力。
- DRY：订单状态推进统一走服务层方法，避免控制器里散落状态修改。
- SOLID：控制器只做参数接收，业务规则集中在 Service，钱包入账继续委托钱包服务。
